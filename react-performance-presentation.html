Performance Optimization

Techniques for faster, leaner, more responsive applications

The goal is actionable takeaways you can apply to your codebase today.

🖥️ Server-Side Rendering

Leverage the server, lighten the client

Let's start with Server-Side Rendering. The core philosophy here is: do as much work as possible on the server so the client receives a fast, lightweight page. This directly impacts First Contentful Paint, Largest Contentful Paint, and SEO.

Why SSR?

  • ⚡ Faster FCP / LCP — HTML arrives ready to render
  • 🔍 Better SEO — crawlers see full content
  • 📦 Reduced client JS — less to download, parse, execute
The three big wins of SSR: First, the browser gets fully-rendered HTML immediately instead of a blank page waiting for JS to execute. This dramatically improves First Contentful Paint and Largest Contentful Paint. Second, search engine crawlers can index your content without executing JavaScript. Third, since rendering happens on the server, you ship less JavaScript to the client — which means faster parse times and lower Time to Interactive.

React Server Components

  • Components execute only on the server
  • Zero JS shipped to client for server components
  • Direct access to databases, file systems, APIs
React Server Components are a game-changer. They run exclusively on the server — no JavaScript for these components ever reaches the browser. This means you can directly access databases, read files, call internal APIs without exposing any of that logic or those dependencies to the client bundle. The component renders to HTML on the server and that's what the user gets. It's the most effective way to reduce client-side JavaScript.

Minimize Serialization

Only pass what the client actually needs


❌ Don't

pass entire user object

✅ Do

pass only { name, avatar }
When data crosses the server-client boundary, it gets serialized as JSON in the page payload. If you pass an entire user object with 50 fields when the client component only needs the name and avatar, you're bloating the page weight unnecessarily. Be intentional about what you pass as props to client components. Think of the server-client boundary like an API — only send what's needed. This reduces HTML size, speeds up hydration, and reduces memory usage on the client.

Parallel Data Fetching

  • Restructure component tree for simultaneous fetches
  • Sibling server components fetch in parallel
  • Eliminates server-side waterfalls
One of the most impactful patterns: if you have a page that needs user data, product data, and recommendations, don't fetch them sequentially. Structure your component tree so these are sibling server components — they'll automatically fetch data in parallel. A sequential waterfall of 3 API calls at 200ms each takes 600ms. In parallel, it's just 200ms. That's a 400ms improvement just from restructuring. Also consider using Promise.all() within a single component when you need multiple data sources.

Suspense + Streaming

  • Wrap async components in <Suspense>
  • Layout renders instantly
  • Data streams in progressively
Suspense with streaming is incredibly powerful. Instead of waiting for ALL data before sending any HTML, the server sends the shell and layout immediately. Users see a meaningful page right away. Then as each data-fetching component resolves, it streams into the page and replaces its fallback. This means your LCP can be near-instant for the page structure, and slower data doesn't block the entire page. In Next.js App Router, this works out of the box with async server components wrapped in Suspense boundaries.

React.cache()

  • Deduplicates repeated server-side calls
  • Scoped to a single request
  • Perfect for: auth checks, DB queries, config
React.cache is a request-level memoization utility. If multiple server components in the same request need to call getUser() or check authentication, React.cache ensures that function only executes once per request. The result is shared across all components that call it. This eliminates redundant database queries and API calls without any manual coordination. It's automatically scoped to the request, so there's no risk of data leaking between users.

When Does SSR Hurt?

  • Heavy computation delays Time to First Byte (TTFB)
  • Large HTML payloads on slow connections
  • Over-hydration — shipping JS to "activate" static content
  • Uncached pages under high traffic = server strain
SSR isn't always the answer. If your page requires heavy computation or slow API calls, TTFB suffers — users stare at a blank screen waiting for the server. Large HTML payloads can actually be slower than a small JS bundle on very slow connections. Over-hydration is a common trap: if you're shipping JavaScript just to add click handlers to otherwise static content, you're paying hydration cost for no benefit. And under high traffic, uncached SSR pages create significant server load. The solution: use streaming to mitigate TTFB issues, static generation where possible, and be selective about what actually needs to be interactive.

Keeping SSR Lightweight

  • Use RSC — eliminate client JS entirely where possible
  • Stream with Suspense — don't block on slow data
  • Minimize serialized props across boundaries
  • Cache aggressively — ISR, React.cache(), HTTP cache
Strategies to keep SSR pages lean: First, maximize use of React Server Components — anything that doesn't need interactivity should be a server component with zero client JS. Second, use Suspense streaming so no single slow data source blocks the page. Third, be disciplined about the server-client boundary — serialize only essential data. Fourth, cache at every layer: use Incremental Static Regeneration for pages that change infrequently, React.cache for request-level dedup, and HTTP caching headers for CDN-level caching. The goal is: fast TTFB, minimal JS, progressive loading.

Impact of 'use client'

  • Creates a boundary — everything below ships JS
  • Push 'use client' as deep as possible
  • Leaf components = smaller client bundles
The 'use client' directive creates a boundary in your component tree. Once you mark a component as a client component, it AND all its children get included in the client bundle. This is why it's critical to push 'use client' as deep as possible in the tree. Don't put it on a page-level wrapper — put it on the small leaf components that actually need interactivity, like a button with an onClick handler or a form with state. A common pattern: have a server component that fetches data and passes it to a small client component that handles the interaction. This keeps the majority of your component tree server-rendered with zero JS.

Data Hydration Impact

  • Server renders HTML → client re-creates React tree
  • Large state = more JS to parse + longer hydration
  • Minimize: use server components, reduce serialized data
Hydration is the process where React on the client "takes over" the server-rendered HTML by attaching event listeners and rebuilding the component tree. The more data and components that need hydrating, the longer this takes — during which the page is visible but not interactive. This is sometimes called the "uncanny valley" of SSR. Large serialized state embedded in the page means more JSON to parse and more components to reconcile. Strategies to reduce hydration cost: use server components (they don't hydrate at all), reduce the data passed to client components, use Suspense to progressively hydrate different sections, and consider partial hydration patterns where only interactive islands hydrate.

📦 Bundle Size

Ship Less JavaScript!

Our second topic: bundle size. The fastest code is code that's never sent to the browser. Every kilobyte of JavaScript has a cost — it needs to be downloaded, parsed, compiled, and executed. On mobile devices, this cost is 2-5x higher. Our goal is to ruthlessly eliminate unnecessary JavaScript from our bundles.

Barrel File Trap

❌ Barrel Import

import { Button } from '@mui/material'

Pulls in 2,225 modules
+4.2s dev startup

✅ Direct Import

import Button from '@mui/material/Button'

Only what you need

Barrel files are index files that re-export everything from a package. When you import from the barrel, bundlers often can't tree-shake effectively, pulling in the entire library. The MUI example is dramatic: importing Button from the barrel pulls in 2,225 modules and adds 4.2 seconds to dev server startup. The fix is simple — import directly from the component's path. In Next.js, you can also add packages to the optimizePackageImports config option, which automatically transforms barrel imports to direct imports at build time. This is already enabled by default for some packages in Next.js 13.5+.

Code Splitting

  • React.lazy() / next/dynamic
  • Heavy components loaded on demand
  • Great for: editors, charts, modals, admin panels
Code splitting means breaking your bundle into smaller chunks that load on demand. Use React.lazy or next/dynamic for components that aren't needed on the initial render — things like rich text editors, charting libraries, modals, or admin-only features. The user only downloads these when they actually navigate to or interact with that feature. In Next.js, the App Router automatically code-splits by route, but you should also split within routes for heavy components. A code editor library might be 500KB — there's no reason to include that in the initial page load.

Defer Third-Party Scripts

  • Analytics, error tracking, logging — load after hydration
  • Use next/dynamic with { ssr: false }
  • Or next/script with strategy="lazyOnload"
Third-party scripts are often the biggest performance offenders. Analytics, error tracking, A/B testing tools — these don't need to be loaded before the user can interact with your page. Use dynamic import with ssr: false to load them only after hydration completes. Or use next/script with the lazyOnload strategy, which loads scripts during browser idle time. The key insight: these tools track user behavior, but users can't generate behavior until the page is interactive, so there's no data loss in deferring them.

Conditional Loading

  • Load modules only when features activate
  • import() inside event handlers or useEffect
  • Feature flags → conditional import()
Take code splitting further with conditional module loading. Don't load a PDF generator until the user clicks "Export PDF". Don't load a map library until the user opens the location tab. Use dynamic import() inside event handlers, useEffect, or behind feature flag checks. This is especially powerful for features that only a percentage of users ever touch — why should everyone pay the download cost? Pattern: const handler = async () => { const { generatePDF } = await import('./pdf-utils'); generatePDF(data); }

Analysis Tools

  • webpack-bundle-analyzer — visual treemap of bundles
  • @next/bundle-analyzer — Next.js integration
  • source-map-explorer — trace bytes to source

Run regularly. Automate in CI to catch regressions.

You can't optimize what you can't measure. Use bundle analysis tools regularly — not just once. webpack-bundle-analyzer generates an interactive treemap showing exactly what's in your bundles and how large each module is. @next/bundle-analyzer wraps this for Next.js projects. source-map-explorer lets you trace every byte back to its source file. Pro tip: add bundle size checks to your CI pipeline. Set a budget (e.g., main bundle < 200KB gzipped) and fail the build if it's exceeded. This prevents gradual bundle bloat from sneaking in through normal development. Consider before/after screenshots in your PRs when making significant changes.

Bundle Size Wins

Before

  • Barrel imports everywhere
  • All scripts loaded upfront
  • Monolithic bundle

After

  • Direct imports + optimizePackageImports
  • Deferred third-party
  • Route-based + feature-based splitting

Typical savings: 30-60% reduction in initial JS

Let's talk real-world impact. Teams that go through this exercise typically see 30-60% reduction in initial JavaScript. One common pattern: a project importing from barrel files across 50+ components, loading all analytics upfront, and serving everything in one chunk. After fixing barrel imports, deferring scripts, and adding code splitting, the initial bundle drops dramatically. I recommend using the bundle analyzer to take before and after screenshots — it makes the improvement visceral and is great for stakeholder buy-in. The dev server also gets significantly faster since it's processing fewer modules on startup.

🧠 React Compiler

Automatic memoization at build time

Topic three: the React Compiler. This is one of the most exciting developments in the React ecosystem. It's a build-time tool that automatically adds memoization to your components — eliminating the need for manual memo(), useMemo(), and useCallback(). Let's look at how it works and what it means for our code.

What It Does

  • Analyzes components at compile time
  • Identifies cacheable values & expressions
  • Inserts memoization automatically
  • Replaces: memo(), useMemo(), useCallback()
The React Compiler analyzes your component code during the build step. It understands the data flow within your components, identifies which values, computations, and callback functions can be safely memoized, and automatically inserts the appropriate caching. This means the compiler does what we've been doing manually with memo, useMemo, and useCallback — but it does it exhaustively and correctly every time. No more guessing which dependencies to include, no more forgotten memoization on expensive computations.

Benefits

  • ✅ No more forgotten useCallback / useMemo
  • ✅ Cleaner code — write natural React
  • ✅ No dependency array bugs
  • ✅ Consistent optimization across entire codebase
The benefits are significant. First, developers no longer need to remember to wrap callbacks and computations — the compiler catches everything. Second, your code becomes cleaner and more readable because you're writing natural React without performance boilerplate. Third, dependency array bugs — one of the most common sources of React bugs — are eliminated because the compiler correctly identifies dependencies. Fourth, optimization is applied consistently across your entire codebase, not just in the components where someone remembered to add it.

Current Status

  • 🧪 Experimental — React 19 / Next.js 15
  • Opt-in via babel-plugin-react-compiler
  • Used in production at Meta (Instagram, Facebook)
The React Compiler is currently experimental but it's being used in production at Meta across Instagram and Facebook. You can opt into it via the babel-plugin-react-compiler. In Next.js 15, there's a config flag to enable it. It's not yet recommended for all production apps, but it's stable enough to experiment with and evaluate. Meta has reported significant performance improvements in their apps — fewer unnecessary re-renders, lower interaction latency, and reduced memory usage.

Rules & Caveats

  • ⚠️ Assumes code follows Rules of React
  • Pure render functions — no side effects in render
  • Violations silently break optimizations
  • Use eslint-plugin-react-compiler to validate
Here's the critical caveat: the React Compiler assumes your code follows the Rules of React. That means pure render functions with no side effects, props and state treated as immutable, and hooks called consistently. If you mutate objects during render, read from external mutable sources without proper synchronization, or have other rule violations, the compiler's optimizations may produce incorrect behavior — and it won't warn you at runtime. The eslint-plugin-react-compiler can help identify violations before you enable the compiler. This is why it's important to audit your code for rule compliance first.

Migration Path

  1. Run ESLint plugin — fix violations
  2. Enable compiler on a subset of files
  3. Validate behavior hasn't changed
  4. Gradually remove manual memo() / useMemo() / useCallback()
The recommended migration path: First, add the eslint-plugin-react-compiler and fix any violations it finds — these are bugs regardless of whether you use the compiler. Second, enable the compiler on a subset of files or components to test. Third, validate that behavior hasn't changed — run your test suite, do manual QA on affected components. Fourth, once you're confident, gradually remove manual memo(), useMemo(), and useCallback() calls. The compiler handles these automatically, and the manual versions just add noise. Before/after: teams report removing hundreds of useMemo/useCallback calls while maintaining or improving performance.

📊 Lighthouse + Metrics

Measure, optimize, verify

Our final topic: Lighthouse and Core Web Vitals. Everything we've discussed so far — SSR, bundle optimization, memoization — these all contribute to better metrics. But you need to measure to know if you're actually improving. Let's look at the key metrics, how to measure them, and additional optimization techniques based on feedback from the team.

Core Web Vitals

LCP
Largest Contentful Paint
< 200ms
INP

Cumulative Layout Shift
Core Web Vitals are the three metrics Google uses for search ranking. LCP measures loading speed — how quickly the largest visible content element renders. Target under 2.5 seconds. INP (which replaced FID) measures responsiveness — the time from user interaction to the next visual update. Target under 200 milliseconds. CLS measures visual stability — how much the page layout shifts unexpectedly. Target under 0.1. These aren't just Google metrics — they directly correlate with user experience and conversion rates. Amazon found that every 100ms of latency cost them 1% in sales.

Lighthouse Workflow

  1. Run audit (DevTools or CI)
  2. Identify bottlenecks
  3. Fix top issues
  4. Re-audit & compare

Categories: Performance, Accessibility, Best Practices, SEO

The Lighthouse workflow is simple but powerful: Run an audit, identify the biggest bottlenecks, fix them, then re-audit to verify improvement. You can run it from Chrome DevTools Lighthouse tab for quick checks, or integrate the lighthouse CLI into your CI pipeline for automated monitoring. Lighthouse reports across four categories: Performance, Accessibility, Best Practices, and SEO. Each gives a score 0-100 with specific, actionable recommendations. For CI integration, consider tools like lighthouse-ci which can set performance budgets and fail PRs that cause regressions.

Managing Re-renders

  • Push state down — colocate state with consumers
  • Split contexts — separate frequently vs rarely changing data
  • Avoid prop drilling through intermediaries
  • Use React.memo() for expensive subtrees (or React Compiler)
Re-renders are one of the biggest performance issues in React apps. Key strategies: First, push state down — if only one component needs a piece of state, don't put it in a parent that causes siblings to re-render. Second, split your contexts — a single context with auth, theme, and user data means every component re-renders when any value changes. Separate them. Third, avoid prop drilling through intermediary components that don't use the props — they re-render just to pass data through. Use composition or context instead. Fourth, for expensive component subtrees that can't be restructured, use React.memo or let the React Compiler handle it.

Prop Hell & Global State

  • Prop hell = excessive prop passing → unnecessary re-renders
  • Every prop change re-renders the entire chain
  • Solution: composition, context, or state management
  • Global state: keep it minimal — only truly global data
Prop hell is when you're passing 15+ props through 5 levels of components. Beyond being unmaintainable, it's a performance problem: when any prop changes, every component in that chain re-renders. The fix is composition — restructure so components receive children rather than data. Or use context/state management for deeply-needed data. For global state: be ruthless about what's truly global. User auth? Global. The currently selected tab in one section? Local. Every piece of data in global state is a potential re-render trigger for every consumer. Use selectors in Zustand/Redux to subscribe to only the specific slices each component needs.

next/image Optimization

  • Automatic format conversion (WebP/AVIF)
  • Responsive srcset — right size for every viewport
  • Lazy loading by default
  • Prevents CLS with automatic dimensions
The next/image component provides huge wins with minimal effort. It automatically converts images to modern formats like WebP and AVIF which are 25-50% smaller. It generates responsive srcset attributes so mobile users download smaller images. It lazy-loads images by default — images below the fold don't load until the user scrolls near them. And it prevents Cumulative Layout Shift by requiring width/height or using fill, so the browser reserves space before the image loads. For your LCP image (hero image), add the priority prop to preload it.

next/script Optimization

  • beforeInteractive — critical scripts (rare)
  • afterInteractive — default, loads after hydration
  • lazyOnload — loads during idle time
  • worker — offloads to web worker (experimental)
next/script gives you fine-grained control over when third-party scripts load. beforeInteractive is for truly critical scripts that must run before the page is interactive — use sparingly. afterInteractive is the default and loads after hydration, which is appropriate for most analytics. lazyOnload defers loading until browser idle time — perfect for non-critical tracking and widgets. The experimental worker strategy offloads script execution to a web worker, keeping the main thread free. Most teams should use lazyOnload for analytics and afterInteractive only for scripts that need DOM access immediately after hydration.

Streaming with Next.js

  • App Router streams by default with loading.js
  • Granular streaming with <Suspense> boundaries
  • Shell renders instantly → data fills in progressively
  • Dramatically improves perceived performance
Next.js App Router supports streaming out of the box. When you add a loading.js file, the layout and loading state are sent immediately while the page component fetches data. For more granular control, wrap individual components in Suspense boundaries — each section of your page streams independently. The user sees the page shell instantly, then content progressively appears as data resolves. This is especially powerful for pages with multiple data sources of varying speeds — the fast data shows immediately, slow data streams in later. Combined with React Server Components, you get the best of both worlds: fast initial render and minimal client JavaScript.

Performance Coding Techniques

  • Virtualize long lists — react-virtual / react-window
  • Debounce expensive operations (search, resize)
  • useTransition for non-urgent updates
  • Avoid inline object/array creation in JSX
Additional coding techniques: Virtualize long lists — if you're rendering 1000+ items, only render what's visible using react-virtual or react-window. Debounce expensive operations like search-as-you-type or resize handlers. Use React's useTransition for state updates that don't need to be immediate — it lets React prioritize urgent updates (like typing) over non-urgent ones (like filtering a large list). Avoid creating new objects and arrays inline in JSX — style={{margin: 10}} creates a new object every render, causing child components to see "new props" and re-render unnecessarily.

🎯 Key Takeaways

  • SSR — Use RSC, stream with Suspense, minimize boundaries
  • Bundle — Direct imports, code split, defer scripts
  • Compiler — Automatic memoization, follow Rules of React
  • Metrics — Measure with Lighthouse, optimize CWV

Measure → Optimize → Verify → Repeat

Let's wrap up with key takeaways. For SSR: maximize React Server Components, use Suspense for streaming, and be intentional about the server-client boundary. For bundle size: use direct imports, aggressively code split, and defer non-critical scripts. For the React Compiler: it automates memoization but requires your code to follow React's rules. For metrics: use Lighthouse and Core Web Vitals as your north star, run audits regularly, and set performance budgets in CI. The overarching principle: measure first, optimize what matters most, verify the improvement, and repeat. Performance is an ongoing practice, not a one-time task. Thank you!
