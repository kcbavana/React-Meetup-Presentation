🚀 Stop Creating Request Waterfalls

❌ Sequential

100ms + 100ms + 100ms

= ~300ms

✅ Parallel

Max(100ms, 100ms, 100ms)

= ~100ms

Contact Info · Subscription · Store Credit — fetched simultaneously

The simplest and highest-impact optimization: don't fetch data sequentially when requests are independent. If you need Contact Info, Subscription status, and Store Credit — those don't depend on each other. Sequentially that's 300ms. In parallel, it's just 100ms — the time of the slowest call. That's a 200ms improvement from just restructuring your data fetching. Use Promise.all, parallel server components, or restructure your component tree so siblings fetch simultaneously.

🚀 Batch by Dependency

Group independent work into waves

Wave 1: User/Session (~100ms)
Wave 2: Profile · Permissions · Purchase History (~100ms)
Wave 3: Contracts · Order Status · Products (~100ms)
Wave 4: Order Feedback (~100ms)

8 calls in ~400ms instead of ~800ms sequential

Some requests unlock other requests — you need the user session before you can fetch their profile or permissions. The pattern is: group requests into waves by dependency. Wave 1 gets the session. Wave 2 fires everything that depends only on the session — profile, permissions, purchase history — all in parallel. Wave 3 fires everything that depends on wave 2 results. Instead of 8 sequential calls at 800ms, you get 4 waves at 400ms. Cut your total time in half just by understanding your dependency graph and batching accordingly.

💾 Caching: Do the Work Once

The next big win: caching. If data hasn't changed, don't fetch it again. Cache stable reads with meaningful keys — for a store page, that's store ID + locale + tenant. Set appropriate TTLs or use event-based revalidation. The benefit is twofold: faster repeat page loads for users, and reduced load on your backend services. Without caching, every navigation calls downstream again — the page waits on data it already saw recently, and backend load grows with navigation churn.

💾 Data Volatility Guide

Store Information

Not all data is equal — set TTLs based on how often data actually changes. Store information like address, hours, name — that changes rarely, cache for hours or a day. Product information — descriptions, specs — changes occasionally, cache for minutes to an hour. Availability and order status change frequently — cache for seconds to a minute at most, or use real-time updates. Payment information — never cache, always fetch fresh for security and accuracy. Match your cache strategy to data volatility.

🖥️ Why SSR?

  • ⚡ Faster FCP / LCP — HTML arrives ready to render
  • 🔍 Better SEO — crawlers see full content
  • 📦 Less client JS — rendering happens server-side
Three big wins of SSR: the browser gets fully-rendered HTML immediately instead of a blank page waiting for JS — dramatically better First and Largest Contentful Paint. Search engines can index without executing JavaScript. And since rendering happens server-side, you ship less JavaScript to the client — faster parse times and lower Time to Interactive.

🖥️ RSC + Streaming

  • React Server Components — zero JS to client
  • <Suspense> — shell renders instantly, data streams in
  • React.cache() — dedup repeated calls per request
React Server Components run exclusively on the server — no JavaScript ever reaches the browser. You can directly access databases and internal APIs. Suspense streaming sends the layout immediately, then content fills in progressively as data resolves. React.cache deduplicates repeated server calls (like auth checks) across components in the same request — the function only executes once, result is shared.

🖥️ Keep SSR Lightweight

  • Push 'use client' to leaf components
  • Minimize serialized props across the boundary
  • Cache: ISR, HTTP headers, React.cache()
The 'use client' directive creates a boundary — everything below ships JS. Push it as deep as possible: small interactive leaf components only, not page wrappers. Only serialize fields the client actually needs — pass name, not the entire user object. Cache aggressively at every layer: ISR for infrequently changing pages, HTTP cache headers for CDN-level caching, React.cache for request-level dedup.

📦 Barrel File Trap

❌ Barrel Import

import { Button } from '@mui/material'

2,225 modules · +4.2s dev startup

✅ Direct Import

import Button from '@mui/material/Button'

Only what you need

Barrel files re-export everything from a package. Importing from the barrel pulls in the entire library. MUI example: 2,225 modules, 4.2 seconds added to dev startup. Fix: import directly from the component path. In Next.js, add packages to optimizePackageImports in the config for automatic barrel-file optimization at build time.

📦 Split & Defer

  • React.lazy() / next/dynamic for heavy components
  • Defer analytics with strategy="lazyOnload"
  • Conditional import() inside event handlers
  • Measure: @next/bundle-analyzer, source-map-explorer

Typical savings: 30-60% reduction in initial JS

Code split heavy components not needed on initial render — editors, charts, modals — with React.lazy or next/dynamic. Defer third-party scripts: analytics doesn't need to load before interaction, use next/script with lazyOnload. For features only some users trigger, use dynamic import() inside event handlers. Always measure with bundle analyzer — add to CI to catch regressions. Teams typically see 30-60% initial JS reduction.

🧠 React Compiler

  • Analyzes components at build time
  • Auto-inserts memoization
  • Replaces manual memo(), useMemo(), useCallback()
The React Compiler analyzes your component code during build. It understands data flow, identifies cacheable values and callbacks, and inserts memoization automatically. It does what we've been doing manually — but exhaustively and correctly every time. No more guessing dependencies, no more forgotten memoization. Write natural React, the compiler optimizes it. Used in production at Meta across Instagram and Facebook.

🧠 Babel Setup

npm install -D babel-plugin-react-compiler eslint-plugin-react-compiler
// babel.config.js
module.exports = {
  plugins: [
    ['babel-plugin-react-compiler', {
      sources: (filename) => filename.includes('src/'),
    }],
  ],
};
Implementation: install babel-plugin-react-compiler and the ESLint plugin. In Babel config, add the plugin with an optional sources filter to target specific directories for incremental rollout. Start with a subset, verify everything works, then expand. The ESLint plugin should be added first — it catches Rules of React violations that would cause incorrect compiled output.

🧠 Next.js Setup

// next.config.js
module.exports = {
  experimental: {
    reactCompiler: true,
  },
};

Next.js 15+ handles Babel config automatically

In Next.js 15+, just set experimental.reactCompiler to true. Next.js configures Babel automatically — no separate babel.config.js needed. You can also pass compilationMode: 'annotation' to only compile components marked with 'use memo' — safest way to start: opt specific components in one at a time.

🧠 ✅ Advantages

  • Eliminates memoization boilerplate
  • Catches optimizations humans miss
  • Zero runtime overhead — build-time only
  • 30-80% fewer unnecessary re-renders
  • No stale closure / dependency array bugs
Advantages: eliminates all manual memoization boilerplate — cleaner code. Developers typically only memoize 20-40% of what should be; the compiler does 100%. Zero runtime overhead since analysis happens at build time. Meta reports 30-80% reduction in unnecessary re-renders. And it eliminates stale closure bugs — a top source of React bugs from forgotten dependencies.

🧠 ❌ Disadvantages

  • Requires strict Rules of React compliance
  • Silent failures — violations compile but misbehave
  • Debugging compiled output is harder
  • Increased build time
  • Still experimental — API may change
Disadvantages: demands strict Rules of React — pure renders, immutable props/state. Silent failures are the scariest: code that violates rules compiles fine but subtly misbehaves — stale data, missed updates, no error. Debugging is harder since compiled output doesn't match source. Adds build time. Still experimental — API could change. Migration: run ESLint plugin first, fix violations, enable on a subset, validate, then gradually remove manual memo/useMemo/useCallback.

Core Web Vitals — the 3 metrics Google uses for ranking. LCP: how fast the largest content renders, target under 2.5s. INP: time from interaction to visual update, target under 200ms. CLS: unexpected layout shifts, target under 0.1. Run Lighthouse in Chrome DevTools or CI via lighthouse CLI. Workflow: audit, identify bottlenecks, fix, re-audit.

📊 Optimization Techniques

  • Re-renders: push state down, split contexts
  • Global state: keep minimal, use selectors
  • next/image: auto WebP/AVIF, lazy load, prevents CLS
  • next/script: lazyOnload for non-critical scripts
  • Virtualize: long lists with react-virtual
Additional techniques: colocate state with consumers, split contexts so one change doesn't re-render everything. Global state should be minimal — use selectors for specific slices. next/image gives automatic format conversion, lazy loading, CLS prevention. next/script with lazyOnload defers non-critical scripts to idle time. Virtualize long lists — only render what's visible. Also: useTransition for non-urgent updates, streaming with Suspense for progressive rendering.
